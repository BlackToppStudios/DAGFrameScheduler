00062 /// @mainpage Directed Acyclic Graph Frame Scheduler.
00063 /// @section goal_sec Goals
00064 /// This library tries to make writing multithreaded software easier by changing the kinds of primitives that
00065 /// multithreaded software is built upon. Several libraries before this have attempted this already.
00066 /// This library is different becuse it focuses on a specific kind of workload and provides the kinds of
00067 /// guarantees that workload needs while sacrificing other guarantees that workload does need.
00068 /// @n @n
00069 /// This attempts to provide a multithreading solution for workloads that must be run in many iterations in
00070 /// a given amount of realtime. Games are an ideal example. Every frame a video game, must update physics
00071 /// simulations, make AI decisions, accept/interpret user input, enforce game rules, perform dynamic I/O
00072 /// and render it to the screen all while maintaining a smooth FrameRate and do that while draining batteries on
00073 /// portable devices (sometimes without even knowing if the device is portable).
00074 /// @n @n
00075 /// This accomplishes those goals by removing the conventional mutlithreading primitives that so many
00076 /// developers have come to fear, loathe or misunderstand. Mutexes, threads, memory fences, thread_local
00077 /// storage, atomic variables, and all the pitfalls that come with them are replaced by a small set of
00078 /// of primitives that provide all the required sophistication a typical multiple iteration application
00079 /// requires. It does this using a new kind of @ref Mezzanine::Threading::WorkUnit "WorkUnit",
00080 /// @ref Mezzanine::Threading::DoubleBufferedResource "Double Buffering", A strong concept of
00081 /// Dependencies and a @ref Mezzanine::Threading::FrameScheduler "FrameScheduler" that uses heuristics
00082 /// to decide how to run it all without exposing needless complexity to the application developer.
00083 ///
00084 /// @section overview_sec Overview
00085 /// The DAGFrameScheduler is a variation on a common multithreaded work queue. It seeks to avoid its pitfalls,
00086 /// such as non-determinism, thread contention and lackluster scalability, while keeping its advantages
00087 /// including simplicity, understandiblity and low overhead.
00088 /// @n @n With this algorithm very few if any
00089 /// calls will need to be made to the underlying system for synchronization, instead it will provide limited
00090 /// deterministic ordering of @ref Mezzanine::Threading::WorkUnit "WorkUnit" execution through a dependency
00091 /// feature. Having the knowledge that one @ref Mezzanine::Threading::WorkUnit "WorkUnit" will complete after
00092 /// another allows for resources to be used without using expensive and complex synchronization mechansisms
00093 /// like @ref Mezzanine::Threading::mutex "mutexes", semaphores, or any of their kin.
00094 /// @n @n The internal work queue is not changed while a frame is executing. Because it is only read, each
00095 /// thread can pick its own work. Synchronization still needs to occur, but it has been moved onto each
00096 /// @ref Mezzanine::Threading::WorkUnit "WorkUnit" it is manages with atomic CPU operations. Like this,
00097 /// contention is less frequent, occurring only when threads simultaneously access the same
00098 /// @ref Mezzanine::Threading::WorkUnit "WorkUnit", and it consumes far less time because atomic operations
00099 /// are CPU instructions instead of Operating System calls.
00100 /// @section broken_sec Broken Algorithms
00101 /// To understand why a new multithreading system is needed, it is helpful to look at other methods
00102 /// of threading that have been used in the past, and understand what they lack or how they aren't ideal
00103 /// for the kinds of work this algorithm is is intended for.
00104 /// @n @n I will use charts that plots possible resource use of a computer across time. Generally time will
00105 /// run accross the top a resources, usually cpus will run down one side.
00106 ///
00107 /// @n @n These threading models models aren't necessarily broken, some of these clearly have a place in software
00108 /// development. None of these are ideal for video games or other tasks that have realtime recurring execution
00109 /// requirements. Many of these require complex algorithms, subtle knowledge or simply aren't performant enough
00110 /// for realtime environments.
00111 /// @subsection broken_Single Single Threaded
00112 /// An application using this threading model is not actually multithreaded at all. However, It has been shown
00113 /// that software can run in a single and get good perfomance. This is benchmark all other threading models
00114 /// get compared too.
00115 /// @n @n There is a term, Speedup ( http://en.wikipedia.org/wiki/Speedup ), which is simply a
00116 /// comparison of the single threaded performance of an algorithm to the mutlithreaded performance. You simply
00117 /// determine how many times more work the multithreaded algorithm does in the same time, or how many times
00118 /// longer the single threaded algorithm takes to the same work. Ideally two threads will be twice as fast
00119 /// (speedup of 2x), and three thread would be three times as fast (3x speedup), and so; this is called linear
00120 /// speedup. In practice there is always some overhead in creating and synchronizing threads, so achieving
00121 /// linear speedup is diffucult.
00122 /// @image html Single.png "Single Threaded Execution - Fig 1."
00123 /// @image latex Single.png "Single Threaded Execution - Fig 1."
00124 /// @image rtf Single.png "Single Threaded Execution - Fig 1."
00125 /// @n @n The DAGFrameScheduler library tries to tailor the threading model to the problem to minimize that
00126 /// overhead. With a single threaded application one thread does all the work and always wastes every other
00127 /// thread.
00128 /// @n @n
00129 /// @subsection broken_Unplanned Unplanned Thread
00130 /// Sometimes someone means well and tries to increase the performance of a single threaded program and tries
00131 /// to add extras threads to increase performance. Sometimes this works, really well, sometimes there is a
00132 /// marginal increase in performance or a significant increase in bugs. If that someone has a good plan
00133 /// then they can usually achieve close to the best speedup possible in the given situation. This is not easy
00134 /// and many cannot do this or do not want to invest the time it would take. If not carefully planned
00135 /// bugs like deadlock ( http://en.wikipedia.org/wiki/Deadlock ) and race conditions ( http://stackoverflow.com/questions/34510/what-is-a-race-condition )
00136 /// can be introduced. Unfortunately no amount of testing can replace this careful planning. Without a
00137 /// complete understanding of how multithreaded software is assembled (a plan) it is not possible to prove
00138 /// that multithreaded software will not hang/freeze or that it will produce the correct results.
00139 /// @n @n
00140 /// Software with no multithreading plan could have just about any kind of execution behavior. Ussually they
00141 /// perform at least slightly better than single threaded versions of the software, but frequently suffer
00142 /// do not utilize all the available resources. Generally performance does not scale well as unplanned
00143 /// software is run on more processors. Frequently there is contention for a specific resource and a
00144 /// thread will wait for that resource longer than is actually need.
00145 /// @image html Unplanned.png "Unplanned Threaded Execution - Fig 2."
00146 /// @image latex Unplanned.png "Unplanned Threaded Execution - Fig 2."
00147 /// @image rtf Unplanned.png "Unplanned Threaded Execution - Fig 2."
00148 /// @n @n The DAGFrameScheduler is carefully planned and completely avoids costly synchronization
00149 /// mechanisms in favor of less costly minimalistic ones. Marking one @ref Mezzanine::Threading::WorkUnit "WorkUnit"
00150 /// as dependent on another allows the reordering of @ref Mezzanine::Threading::WorkUnit "WorkUnits" so that
00151 /// some @ref Mezzanine::Threading::WorkUnit "WorkUnit" can be executed with no thread waiting or blocking.
00152 /// @n @n
00153 /// @subsubsection broken_TaskPerThread One Task Per Thread
00154 /// A common example of poor planning is the creation of one thread for each task in a game. Despite
00155 /// being conceptually simple, performance of systems designed this was is poor due to synchronization
00156 /// and complexities that synchronization requires.
00157 /// @subsection broken_ConventionWorkQueue Convention Work Queue/Thread Pools
00158 /// Conventional work queues and thread pools are well known and robust way to increase the throughput of
00159 /// of an application. These are ideal solutions for many systems, but not games.
00160 /// @n @n In work conventional workqueues all of the work is broken into a number of small thread-safe
00161 /// units. As these units are created they are stuffed into a queue and threads pull out units of work
00162 /// as it completes other units it has started. This simple plan has many advantages. If there is work
00163 /// to do, then at least one thread will be doing, and usually more threads will be working; this is good
00164 /// for games and the DAGFrameScheduler mimics it. If the kind of work is unknown when the software is
00165 /// written heuristics and runtime decisions can create the kind of units of work that are required. This
00166 /// is not the case with games and the others kinds of software this library caters to, so changes can
00167 /// be made that remove the problems this causes. One such drawback is that a given unit of work never
00168 /// knows if another is running or has completed, and must therefor make some pessimistic assumptions.
00169 /// @image html Threadpool.png "Convention Work Queue/ThreadPools - Fig 3."
00170 /// @image latex Threadpool.png "Convention Work Queue/ThreadPools - Fig 3."
00171 /// @image rtf Threadpool.png "Convention Work Queue/ThreadPools - Fig 3."
00172 /// @n @n Common synchronization mechanisms like mutexes or semaphores block the thread for an unknown
00173 /// amount of time, and are required by the design of workqueues. There are two times this is required.
00174 /// The first time is whenever a work unit is acquired by a thread, a mutex (or similar) must be used
00175 /// to prevent other threads from modifying the queue as well. This impacts scalability, but can be
00176 /// circumvented to a point. Common ways to work around this try to split up the work queue
00177 /// pre-emptively, or feed the threads work units from varying points in the queue. The
00178 /// DAGFrameScheduler moves the synchronizantion onto each work to greatly reduce the contention as
00179 /// more workunits are added.
00180 /// @n @n The other, and less obvious, point of contention that has not be circumvented in a
00181 /// satisfactory way for games is the large of amount of synchronization required between units of
00182 /// work that must communicate. For example, there may be hundreds of thousands of pieces of data
00183 /// that must be passed from a system into a 3d rendering system. Apply mutexes to each would slow
00184 /// execution an impossibly long time (if it didn't introduce deadlock), while more coarse grained
00185 /// lock would prevent large portions of physics and rendering from occurring at the time causing
00186 /// one or both of them to wait/block. A simple solution would be to run physics before graphics,
00187 /// but common work queues do not provide good guarantees in this regard.
00188 /// @n @n The DAGFrameScheduler was explicitly designed to provide exactly this guarantee. If the
00189 /// physics @ref Mezzanine::Threading::WorkUnit "WorkUnit" is add to the graphics
00190 /// @ref Mezzanine::Threading::WorkUnit "WorkUnit" with
00191 /// @ref Mezzanine::Threading::WorkUnit::AddDependent() "AddDependent(WorkUnit*)" then it will
00192 /// always be run before the graphics workunit in a given frame.
