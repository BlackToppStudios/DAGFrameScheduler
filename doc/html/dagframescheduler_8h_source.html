00077 /// @mainpage Directed Acyclic Graph Frame Scheduler.
00078 /// @section goal_sec Goals
00079 /// This library tries to make writing multithreaded software easier by changing the kinds of primitives that
00080 /// multithreaded software is built upon. Several libraries before this have attempted this already.
00081 /// This library is different becuse it focuses on a specific kind of workload and provides the kinds of
00082 /// guarantees that workload needs while sacrificing other guarantees that workload does need.
00083 /// @n @n
00084 /// This attempts to provide a multithreading solution for workloads that must be run in many iterations in
00085 /// a given amount of realtime. Games are an ideal example. Every frame a video game, must update physics
00086 /// simulations, make AI decisions, accept/interpret user input, enforce game rules, perform dynamic I/O
00087 /// and render it to the screen all while maintaining a smooth FrameRate and do that while minimizing drain
00088 /// batteries on portable devices (sometimes without even knowing if the device is portable).
00089 /// @n @n
00090 /// This library accomplishes those goals by removing the conventional mutlithreading primitives that so many
00091 /// developers have come to fear, loathe or misunderstand. Mutexes, threads, memory fences, thread_local
00092 /// storage, atomic variables, and all the pitfalls that come with them are replaced by a small set of
00093 /// of primitives that provide all the required sophistication a typical multi-threaded application
00094 /// requires. It does this using a new kind of @ref Mezzanine::Threading::iWorkUnit "iWorkUnit",
00095 /// @ref Mezzanine::Threading::DoubleBufferedResource "Double Buffering", A strong concept of
00096 /// Dependencies and a @ref Mezzanine::Threading::FrameScheduler "FrameScheduler" that uses heuristics
00097 /// to decide how to run it all without exposing needless complexity to the application developer.
00098 ///
00099 /// @section overview_sec Overview
00100 /// The DAGFrameScheduler is a variation on a common multithreaded work queue. It seeks to avoid its pitfalls,
00101 /// such as non-determinism, thread contention and lackluster scalability, while keeping its advantages
00102 /// including simplicity, understandiblity and low overhead.
00103 /// @n @n
00104 /// With this algorithm very few if any
00105 /// calls will need to be made to the underlying system for synchronization in the actual work to be performed.
00106 /// Instead, this library will provide limited
00107 /// deterministic ordering of @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" execution through a dependency
00108 /// feature. Having the knowledge that one @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" will complete after
00109 /// another allows for resources to be used without using expensive and complex synchronization mechansisms
00110 /// like @ref Mezzanine::Threading::Mutex "mutexes", semaphores, or even an
00111 /// @ref Mezzanine::Threading::AtomicCompareAndSwap32 "Atomic Compare And Swaps". These primitives are provided
00112 /// to allow use of this library in advanced ways for developers who are already familiar with
00113 /// multithreaded systems.
00114 /// @n @n
00115 /// The internal work queue is not changed while a frame is executing. Because it is only read, each
00116 /// thread can pick its own work. Synchronization still needs to occur, but it has been moved onto each
00117 /// @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" it is manages this with atomic CPU operations. Like this,
00118 /// contention is less frequent, occurring only when threads simultaneously attempt to start the same
00119 /// @ref Mezzanine::Threading::iWorkUnit "iWorkUnit", and it consumes far less time because atomic operations
00120 /// are CPU instructions instead of Operating System calls. This is managed by the library, so individual
00121 /// @ref Mezzanine::Threading::iWorkUnit "iWorkUnit"s do not need to worry synchronization beyond telling
00122 /// each @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" about its data dependencies and making sure
00123 /// all the @ref Mezzanine::Threading::iWorkUnit "iWorkUnit"s added to a
00124 /// @ref Mezzanine::Threading::FrameScheduler "FrameScheduler".
00125 ///
00126 /// @section broken_sec Broken Algorithms
00127 /// To understand why a new multithreading system is needed, it is helpful to look at other methods
00128 /// of threading that have been used in the past. This can give us an understanding of what they lack
00129 ///  or how they aren't ideal for the kinds of work this algorithm is intended for. This overview is
00130 /// intentionally simplified. There are variations on many of these algorithms that can fix some of
00131 /// the problems presented. Despite these workarounds there are fundamental limitations that prevent
00132 /// these algorithms from being ideal for the video games and similar tasks.
00133 /// These threading models aren't necessarily broken, some of these clearly have a place in software
00134 /// development. Many of these require complex algorithms, require subtle knowledge or simply aren't
00135 /// performant enough for realtime environments.
00136 /// @n @n
00137 /// I will use charts that plot possible resource use of a computer across time. Generally time will
00138 /// run accross the top a resources, usually CPUs will run down one side. Most of these algorithms have a
00139 /// concept of tasks or workunits, these are just piece of work with a distinct begining and end. The
00140 /// width of a piece of work loosely represents the execution time (the names are just for show and not
00141 /// related to anything real).
00142 /// @subsection broken_Single Single Threaded
00143 /// An application using this threading model is not actually multithreaded at all. However, It has been shown
00144 /// that software can run in a single and get good perfomance. This is benchmark all other threading models
00145 /// get compared too.
00146 /// @n @n
00147 /// There is a term, Speedup ( http://en.wikipedia.org/wiki/Speedup ), which is simply a
00148 /// comparison of the single threaded performance of an algorithm to the mutlithreaded performance. You simply
00149 /// determine how many times more work the multithreaded algorithm does in the same time, or how many times
00150 /// longer the single threaded algorithm takes to the same work. Ideally two threads will be twice as fast
00151 /// (speedup of 2x), and three thread would be three times as fast (3x speedup), and so; this is called linear
00152 /// speedup. In practice there is always some overhead in creating and synchronizing threads, so achieving
00153 /// linear speedup is diffucult.
00154 /// @image html Single.png "Single Threaded Execution - Fig 1."
00155 /// @image latex Single.png "Single Threaded Execution - Fig 1."
00156 /// @image rtf Single.png "Single Threaded Execution - Fig 1."
00157 /// @n @n The DAGFrameScheduler library tries to tailor the threading model to the problem to minimize that
00158 /// overhead. With a single threaded application one thread does all the work and always wastes every other
00159 /// thread, but there is no overhead if the system only has one thread.
00160 /// @n @n
00161 /// @subsection broken_Unplanned Unplanned Thread
00162 /// Sometimes someone means well and tries to increase the performance of a single threaded program and tries
00163 /// to add extra threads to increase performance. Sometimes this works, really well, sometimes there is a
00164 /// marginal increase in performance or a significant increase in bugs. If that someone has a good plan
00165 /// then they can usually achieve close to the best speedup possible in the given situation. This is not easy
00166 /// and many cannot do this or do not want to invest the time it would take. If not carefully planned
00167 /// bugs like deadlock ( http://en.wikipedia.org/wiki/Deadlock ) and race conditions
00168 /// ( http://stackoverflow.com/questions/34510/what-is-a-race-condition )
00169 /// can be introduced. Unfortunately no amount of testing can replace this careful planning. Without a
00170 /// complete understanding of how multithreaded software is assembled (a plan) it is not possible to prove
00171 /// that multithreaded software will not hang/freeze or that it will produce the correct results.
00172 /// @n @n
00173 /// Software with no multithreading plan could have just about any kind of execution behavior. Usually
00174 /// unplanned software performs at least slightly better than single threaded versions of the software, but
00175 /// frequently does not utilize all the available resources. Generally performance does not scale well as
00176 /// unplanned software is run on more processors. Frequently, there is contention for a specific resource and
00177 /// a thread will wait for that resource longer than is actually need.
00178 /// @image html Unplanned.png "Unplanned Threaded Execution - Fig 2."
00179 /// @image latex Unplanned.png "Unplanned Threaded Execution - Fig 2."
00180 /// @image rtf Unplanned.png "Unplanned Threaded Execution - Fig 2."
00181 /// @n @n
00182 /// The DAGFrameScheduler is carefully planned and completely avoids costly synchronization
00183 /// mechanisms in favor of less costly minimalistic ones. Marking one @ref Mezzanine::Threading::iWorkUnit "iWorkUnit"
00184 /// as dependent on another allows the reordering of @ref Mezzanine::Threading::iWorkUnit "iWorkUnits" so that
00185 /// some @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" can be executed with no thread waiting or blocking.
00186 /// @n @n
00187 /// @subsubsection broken_TaskPerThread One Task Per Thread
00188 /// A common example of poor planning is the creation of one thread for each task in a game. Despite
00189 /// being conceptually simple, performance of systems designed this was is poor due to synchronization
00190 /// and complexities that synchronization requires.
00191 /// @subsection broken_ConventionWorkQueue Convention Work Queue/Thread Pools
00192 /// Conventional work queues and thread pools are well known and robust way to increase the throughput of
00193 /// of an application. These are ideal solutions for many systems, but not games.
00194 /// @n @n
00195 /// In conventional workqueues all of the work is broken into a number of small thread-safe
00196 /// units. As these units are created they are stuffed into a queue and threads pull out units of work
00197 /// as it completes other units it has started. This simple plan has many advantages. If there is work
00198 /// to do, then at least one thread will be doing some, and usually more threads will be working; this is
00199 /// good for games and the DAGFrameScheduler mimics it. If the kind of work is unknown when the software is
00200 /// written heuristics and runtime decisions can create the kind of units of work that are required. This
00201 /// is not the case with games and the others kinds of software this library caters to, so changes can
00202 /// be made that remove the problems this causes. One such drawback is that a given unit of work never
00203 /// knows if another is running or has completed, and must therefor make some pessimistic assumptions.
00204 /// @image html Threadpool.png "Convention Work Queue/ThreadPools - Fig 3."
00205 /// @image latex Threadpool.png "Convention Work Queue/ThreadPools - Fig 3."
00206 /// @image rtf Threadpool.png "Convention Work Queue/ThreadPools - Fig 3."
00207 /// @n @n
00208 /// Common synchronization mechanisms like mutexes or semaphores block the thread for an unknown
00209 /// amount of time, and are required by the design of workqueues. There are two times this is required.
00210 /// The first time is whenever a work unit is acquired by a thread, a mutex (or similar) must be used
00211 /// to prevent other threads from modifying the queue as well. This impacts scalability, but can be
00212 /// circumvented to a point. Common ways to work around this try to split up the work queue
00213 /// pre-emptively, or feed the threads work units from varying points in the queue. The
00214 /// DAGFrameScheduler moves the synchronizantion onto each work to greatly reduce the contention as
00215 /// more workunits are added.
00216 /// @n @n
00217 /// The other, and less obvious, point of contention that has not be circumvented in a
00218 /// satisfactory way for games is the large of amount of synchronization required between units of
00219 /// work that must communicate. For example, there may be hundreds of thousands of pieces of data
00220 /// that must be passed from a system into a 3d rendering system. Apply mutexes to each would slow
00221 /// execution an impossibly long time (if it didn't introduce deadlock), while more coarse grained
00222 /// lock would prevent large portions of physics and rendering from occurring at the time causing
00223 /// one or both of them to wait/block. A simple solution would be to run physics before graphics,
00224 /// but common work queues do not provide good guarantees in this regard.
00225 /// @n @n
00226 /// The DAGFrameScheduler was explicitly designed to provide exactly this guarantee. If the
00227 /// physics @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" is added to the graphics
00228 /// @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" with
00229 /// @ref Mezzanine::Threading::iWorkUnit::AddDependency() "AddDependency(WorkUnit*)" then it will
00230 /// always be run before the graphics workunit in a given frame. The drawback of this is that it
00231 /// is more difficult to make runtime creation of workunits (It is possible but it cannot be done
00232 /// during any frame execution), but completely removes the locking
00233 /// mechanisms a conventional work queues. The DAGFrameScheduler has traded one useless feature
00234 /// for a useful guarantee.
00235 ///
00236 /// @section algorithm_sec The Algorithm
00237 /// When first creating the DAGFrameScheduler it was called it "Dagma-CP". When describing it this
00238 /// phrase "Directed Acyclic Graph Minimal Assembly of Critical Path" was used. If you are lucky
00239 /// enough to knows what all those terms mean when assembled this way they are very descriptive. For
00240 /// rest of us the algorithm tries to determine what is the shortest way to execute the work in a
00241 /// minimalistic way using a mathematical graph. The graph is based on what work must done each
00242 /// before what other work each frame and executing it. All the work in this graph will have a
00243 /// location somewhere between the beginning and end, and will never circle around back so it can
00244 /// be called acyclic.
00245 /// @n @n
00246 /// This algorithm was designed with practicality as the first priority. To accomodate and integrate
00247 /// with a variety of other algorithms and system a variety of Work Units are provided. New classes
00248 /// can be created that inherit from these to allow them to be in the scheduler where they will work
00249 /// best.
00250 ///     @li @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" - The interface for a common workunit.
00251 /// These work units will be executed once be frame after all their dependencies have completed.
00252 /// These are also expected to complete execution in a relatively brief period of time compared to
00253 /// the length of a frame, and create no threads while doing so.
00254 ///     @li @ref Mezzanine::Threading::DefaultWorkUnit "DefaultWorkUnit" - A simple implementation
00255 /// of an @ref Mezzanine::Threading::iWorkUnit "iWorkUnit". This may not be suitable for every
00256 /// use case, but it should be suitable for most. Just one function for derived classes to
00257 /// implement, the one that does actual work.
00258 ///     @li @ref Mezzanine::Threading::iAsynchronousWorkUnit "iAsynchronousWorkUnit" - Intended to
00259 /// allow loading of files and streams even after the framescheduler has paused. Work units are
00260 /// to spawn one thread and manage it without interfering with other execution. DMA, and other
00261 /// hardware coprocessors are expected to be utilized to their fullest to help accomplish this.
00262 ///     @li @ref Mezzanine::Threading::MonopolyWorkUnit "MonopolyWorkUnit" - These are expected
00263 /// to monopolize cpu resources at the beginning of each frame. This is ideal when working with
00264 /// other systems, forexample a phsyics system like Bullet3D. If the calls to a physics system are
00265 /// wrapped in a @ref Mezzanine::Threading::MonopolyWorkUnit "MonopolyWorkUnit" then it will be
00266 /// given full opportunity to run before other work units.
00267 ///
00268 /// Once all the @ref Mezzanine::Threading::MonopolyWorkUnit "MonopolyWorkUnit"s are done then the
00269 /// @ref Mezzanine::Threading::FrameScheduler "FrameScheduler" class instance spawns or activates
00270 /// a number of threads based on a simple heuristic. This heuristic is the way work units are sorted
00271 /// in preparation for execution. To understand how these are sorted, the dependency system needs to
00272 /// be understood.
00273 /// @n @n
00274 /// Most other work queues do not provide any guarantee about the order work will be executed in.
00275 /// This means that each piece of work must ensure its own data integrity using synchronization
00276 /// primitives like mutexes and semaphores to protect from being corrupted by multithread access. In
00277 /// most cases these should be removed and one of any two work units that must read/write the data
00278 /// must depend on the other. This allows the code in the workunits to be very simple even if it
00279 /// needs to use a great deal of data other work units may also consume or produce.
00280 /// @n @n
00281 /// Once all the dependencies are in place for any synchronization that has been removed, a
00282 /// @ref Mezzanine::Threading::FrameScheduler "FrameScheduler" can be created and started. At
00283 /// runtime this create a reverse dependency graph, a dependent graph. This is used for determine
00284 /// which work units are the most depended on. For each work unit s simple count of how many work
00285 /// units cannot start until has been completed is generated. The higher this number the earlier
00286 /// the work unit will be executed in a frame. Additionally workunits that take longer to execute
00287 /// will be prioritized ahead of work units that are faster.
00288 /// @n @n
00289 /// Here is a chart that provides an example of this re-factoring and the runtime sorting process:
00290 /// @image html DAGSorting.png "DAG WorkSorting - Fig 4."
00291 /// @image latex DAGSorting.png "DAG WorkSorting - Fig 4."
00292 /// @image rtf DAGSorting.png "DAG WorkSorting - Fig 4."
00293 /// @n @n
00294 /// There are several advantages this sorting provides that are not immediately obvious. It separates
00295 /// the scheduling from the execution allowing, the relatively costly sorting process to be executed
00296 /// only when work units are added, removed or changed. In theory the sorting could be done in a
00297 /// work then updated when the frame is complete. Prioritizing Workunits that take longer to run
00298 /// should help insure the shortest critical path is found by minimizing how often dependencies
00299 /// cause threads to wait for more work.
00300 /// @n @n
00301 /// Each thread queries the
00302 /// @ref Mezzanine::Threading::FrameScheduler "FrameScheduler" for the next piece of work...asdf
00303 /// @image html DAGThreads.png "DAG threads - Fig 5."
00304 /// @image latex DAGThread.png "DAG threads - Fig 5."
00305 /// @image rtf DAGThreads.png "DAG threads - Fig 5."
00306 /// @n @n
00307 /// Some work must be run on specific threads, such as calls to underlying devices (for example,
00308 /// graphics cards using Directx or OpenGL). These @ref Mezzanine::Threading::iWorkUnit "iWorkUnit"s
00309 /// are put into a different listing where only the main thread will attempt to execute them. Other
00310 /// than running these, and running these first, the behavior of the main thread is very similar to
00311 /// other threads. Once a @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" has been completed the
00312 /// thread will query the @ref Mezzanine::Threading::FrameScheduler "FrameScheduler" for more work.
00313 /// Because the @ref Mezzanine::Threading::FrameScheduler "FrameScheduler" is never modified during
00314 /// a frame there is no need for synchronization with it specifically, this avoids a key point of
00315 /// contention that reduces scaling. Instead the synchronization is performed with each
00316 /// @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" and is an
00317 /// @ref Mezzanine::Threading::AtomicCompareAndSwap32 "Atomic Compare And Swap" operation to maximize
00318 /// performance.
00319 /// @n @n
00320 /// Even much of the @ref Mezzanine::Threading::FrameScheduler "FrameScheduler"'s work is performed
00321 /// in @ref Mezzanine::Threading::iWorkUnit "iWorkUnit"s, such as log aggregation and certain functions
00322 /// that must be performed each frame.
00323 /// @ref Mezzanine::Threading::iAsynchronousWorkUnit "iAsynchronousWorkUnit"s continue to run in a thread
00324 /// beyond normal scheduling and are intended to will consume fewer CPU resources and more IO resources.
00325 /// For example loading a large file or listening for network traffic. These will be normal
00326 /// @ref Mezzanine::Threading::iWorkUnit "iWorkUnit"s in most regards and will check on the asynchronous
00327 /// tasks they manage each frame when they run as a normally scheduled.
00328 /// @n @n
00329 /// If a thread should run out of work because all the work is completed the frame will pause until it
00330 /// should start the next frame. This pause length is calulated using a runtime configurable value on
00331 /// the @ref Mezzanine::Threading::FrameScheduler "FrameScheduler". If a thread has checked every
00332 /// @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" and some are still not executing, but could not
00333 /// be started because of incomplete dependencies the thread will simply iterate over every
00334 /// @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" in the
00335 /// @ref Mezzanine::Threading::FrameScheduler "FrameScheduler" until the dependencies of one are
00336 /// met and allows one to be executed. This implicitly guarantees that at least one thread will
00337 /// always do work, and if dependencies chains are kept short then it is more likely that several
00338 /// threads will advance.
00339 /// @n @n
00340 /// The @ref Mezzanine::Threading::iWorkUnit "iWorkUnit" classes are designed to be inherited from
00341 /// and inserted into a @ref Mezzanine::Threading::FrameScheduler "FrameScheduler" which will
00342 /// manage their lifetime and execute them when requested via
00343 /// @ref Mezzanine::Threading::FrameScheduler::DoOneFrame() "FrameScheduler::DoOneFrame()".
