00061         /// @def TimingCostAllowanceGap
00062         /// @brief This is a configuration settings to tune the precision of timing at compile time.
00063         /// @details Each frame that is executed has a desired length of time as determined by the Desired
00064         /// Frame Rate that can set using @ref Mezzanine::Threading::FrameScheduler::SetFrameRate "FrameScheduler::SetFrameRate()"
00065         /// or @ref Mezzanine::Threading::FrameScheduler::SetFrameLength "FrameScheduler::SetFrameLength()".
00066         /// If execution of a frame finishes before it has consumed its portion of a second, then work is paused
00067         /// the desired length has elapsed. This means that if there is too much work each frame that the frame rate
00068         /// will suffer.
00069         /// @n @n
00070         /// The timing of frame lengths with this algorithm is not perfect. Each frame ends a certain
00071         /// amount of microseconds early or late. This inaccuracy can be introduced by the system's timer
00072         /// resolution, the time it takes to lookup the time, or by the work that must be done outside of
00073         /// the of the timed work or pause periods to aggregate data. On Ubuntu 12.04 x64 has a simple
00074         /// timer with a single microsecond resolution, this algorithm tends to drift less than 2 microseconds per
00075         /// frame. Some other algorithms are more precise, and others are simpler, this is a balance between
00076         /// those. If this is not acceptable @ref Mezzanine::GetTimeStamp() "GetTimeStamp()" and
00077         /// @ref Mezzanine::Threading::FrameScheduler::WaitUntilNextThread() "FrameScheduler::WaitUntilNextThread()" could be
00078         /// modified to adjust this.
00079         /// @n @n
00080         /// The pause that is inserted each frame is not simply calculated by subtracting the current time
00081         /// from the target time once the work is done each frame. A small
00082         /// @ref Mezzanine::Threading::FrameScheduler::TimingCostAllowance "TimingCostAllowance" is
00083         /// subtracted from the pause before the next frame begins. This allowance allows for the compensation of
00084         /// variances in scheduling from the underlying system, rounding errors of the system clock and
00085         /// the execution of the time tracking itself. As part of this time tracking the timing cost allowance
00086         /// is adjusted. If the allowance was too large, therefor making the frame too short, then it is made
00087         /// longer for the next frame. If the allowance is too small it is made larger for the next frame. It does
00088         /// seem possible to track this length within a single frame without increasing the complexity a
00089         /// significant amount.
00090         /// @n @n
00091         /// Rather than increase the complexity of the algorithm when the default level of accuracy is
00092         /// suitable for most purposes, a manual compile time adjustment has been added. When comparing the length
00093         /// of the frame after execution and frame pause have occurred, the timing allowance is made smaller or
00094         /// larger based on how close the result is to the desired time. If the frame was not exactly as fast as
00095         /// it should have been, then the allowance is adjusted. Frames tend to take slightly longer than the
00096         /// target length. To compensate for this if the frame executed faster by a predefined amount or less
00097         /// then the timing allowance is not adjusted. This gap in the adjustment of the timing allowance could
00098         /// allow adjustment on different platforms in the precision of this algorithm without incurring extra
00099         /// complexity in timing. By default this is set to 10 microseconds.
00100         /// @ref Mezzanine::Threading::FrameScheduler::GetFrameLength "FrameScheduler::GetFrameLength()"
00101         #define  TimingCostAllowanceGap 10
